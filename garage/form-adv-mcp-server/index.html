Form ADV MCP Server

The data engine behind the ADV tools: one pipeline that turns three messy SEC datasets into a queryable store, a seven-tool MCP server, and a pair of locally fine-tuned models.

The front-end ADV tools on this site all kept running into the same wall: a clean interface is only as good as the data layer underneath it. So I went down a level and built the layer itself — the pipeline, the database, the server, and a model that actually knows the data.

The Problem: Three Datasets That Never Talk

Everything you’d want to know about an investment adviser is technically public, but it’s scattered across three filings that were never designed to be read together. Form ADV Part 1 is structured registration data — AUM, offices, headcount, and the Item 11 disclosure checkboxes. Part 2A brochures are narrative PDFs where firms describe strategy, fees, and conflicts in plain English. Form 13F is a quarterly dump of institutional equity holdings. They live in different formats, on different schedules, behind different SEC endpoints.

Stitching them together by hand is the tax every analyst pays. And even once you’ve stitched them, none of it is shaped for the way I actually want to work now — asking an AI agent a question and having it pull the answer straight from the source.

The Solution: A Data Engine, Not Another UI

Form ADV MCP Server is the part of the system that does the unglamorous work. It ingests all three SEC sources, joins them on the CRD number — the one identifier that runs through every filing — and lands them in a single queryable store. Then it puts a clean interface on top of that store aimed not at a human clicking around, but at an AI client asking questions through the Model Context Protocol.

The result is that an agent in Claude, Cursor, or any MCP-aware client can search 56,000+ firms, pull a single adviser’s full record, list its disclosures, read its brochure, and inspect its 13F holdings — without me writing a scraper for each task.

How It Works

  • The pipeline: A polite, resumable downloader pulls the SEC bulk files (rate-limited and identified, respecting SEC fair-access rules), unzips them, and streams Part 1, the brochures, and the 13F holdings through a pandas transform layer into the database. 13F holdings load in 10k-row chunks because there are 6.75 million of them.
  • The database: A 1.2GB SQLite build for local development with FTS5 full-text search, and a Postgres path for production. Same schema, same loader — the store is just swapped underneath.
  • The MCP server: Seven tools built on FastMCP — search_advisers, get_adviser, get_disclosures, get_brochure, get_13f_holdings, stats, and a read-only query_sql escape hatch for ad-hoc questions.
  • The models: Two TinyLlama-1.1B models fine-tuned with QLoRA (4-bit quantization, LoRA adapters) on a single 6GB laptop GPU — one general Q&A adapter and one specialized for reading disclosures.
  • The dashboard: A FastAPI “AI Studio” that serves the search portal, runs the local model, and tracks training, all packaged behind Docker Compose.

The Architecture

The whole thing is deliberately local-first. SQLite plus FTS5 means the entire 56K-firm universe is searchable on a laptop with no server to stand up, and the same pandas transform code paths into PostgreSQL when it needs to scale. The MCP layer is FastMCP over Python, so the tools are just typed functions — easy to reason about and easy to extend.

The interesting risk was the model. Rather than call out to a hosted LLM, I wanted something that knew this dataset and ran on hardware I own. QLoRA makes that possible: quantize the base TinyLlama to 4-bit, train small LoRA adapters on Q&A pairs generated from the live database, and you get a model that answers ADV questions on a 6GB GPU. It’s small and it’s specialized — which is exactly the point.

I wrote up the full data analysis that came out of this build separately — the scale, the geography, the disclosure landscape, and what institutions actually hold — in The Universe of Form ADV.

Roadmap

The engine works end-to-end today, but “almost built” is the honest status. The next moves are about making it live rather than local: ingestion that auto-tracks the SEC’s monthly and quarterly releases, a hosted MCP endpoint so nobody has to clone a repo, semantic search across the brochure text, change-tracking and alerts when a firm’s filing shifts, and retrieval-augmented answers that keep the fine-tuned models grounded in real rows instead of their own memory. The goal is for this to quietly power the rest of the ADV tools — the layer you never see but the reason the rest feels fast.
